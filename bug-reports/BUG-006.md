# BUG-006 — Leading binary operator yields `NaN` instead of consistent `Error`

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-006                                            |
| **Status**   | Open                                               |
| **Severity** | Medium                                             |
| **Priority** | P2                                                 |
| **Component**| Parser — error reporting                           |

## Summary

A trailing operator (`1+`) correctly yields `Error`, but a leading
operator (`+5`) yields `NaN`. The two malformed-expression cases should
follow the same code path; the inconsistency confuses users and makes
the calculator look poorly tested.

## Steps to reproduce

1. Open the calculator.
2. Press `+`, then `5`, then `=`.

## Expected behaviour

Display shows `Error` (matching the existing `1+ → Error` pathway).

## Actual behaviour

Display shows `NaN`.

## Evidence

| Type | Path |
|--|--|
| Failing test | `tests/functional/test_digits_and_decimal.py::TestClearAndErrors::test_leading_operator_must_yield_error_consistent_with_trailing` |

## Root cause hypothesis

The recursive-descent `_factor` rule throws on missing operand for the
trailing case (caught by `calculate`'s `try/catch` and converted to
`Error`). But for the *leading* case, the LHS factor is consumed and
returns `undefined`/`NaN`, which then propagates through `+` arithmetic
without throwing — landing as `NaN` in `display.value`.

## Suggested fix

In `calculate`, after evaluation, also normalise non-finite results
into `Error`:

```js
const r = evaluateExpression(display.value);
display.value =
    (r === undefined || Number.isNaN(r) || !Number.isFinite(r))
      ? 'Error'
      : r;
```

(This single change also subsumes BUG-005, BUG-008 and BUG-011.)

## Severity justification

**Medium.** Inconsistent error pathways are a known source of user
frustration but do not cause data loss or wrong arithmetic. The fix
above also resolves several other bugs as a side effect, which raises
the priority slightly.
