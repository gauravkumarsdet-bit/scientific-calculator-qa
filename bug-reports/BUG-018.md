# BUG-018 — No keyboard input (WCAG 2.1.1 violation)

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-018                                            |
| **Status**   | Open                                               |
| **Severity** | Medium                                             |
| **Priority** | P2                                                 |
| **Component**| Accessibility / UX — input handling                |

## Summary

The calculator has no `keydown` / `keypress` handler. Pressing digit
keys, operator keys, `Enter` (for `=`), `Backspace`, or `Escape` (for
clear) on the keyboard has no effect. A scientific calculator is
*the* canonical example of a keyboard-friendly tool — its absence
here is a productivity problem and a WCAG 2.1.1 (Keyboard) failure.

## Steps to reproduce

1. Open the calculator.
2. Click anywhere on the page so it has focus.
3. Type `7+2` on the keyboard.

## Expected behaviour

The display updates as if those keys had been clicked.

## Actual behaviour

No effect — the display remains empty.

## Evidence

| Type | Path |
|--|--|
| Failing test | `tests/ux/test_ux_and_accessibility.py::TestKeyboardInput::test_typing_a_digit_on_keyboard_updates_the_display` |

## Suggested fix

Add a single document-level `keydown` listener:

```js
document.addEventListener('keydown', (e) => {
  if (/^[0-9]$/.test(e.key)) return append(e.key);
  if (['+', '-', '*', '/', '(', ')', '.'].includes(e.key)) return append(e.key);
  if (e.key === 'Enter' || e.key === '=') return calculate();
  if (e.key === 'Escape')                  return clearDisplay();
  if (e.key === 'Backspace') {
    e.preventDefault();
    display.value = display.value.slice(0, -1);
  }
});
```

## Severity justification

**Medium.** WCAG 2.1.1 makes keyboard operability a Level A success
criterion — non-negotiable for accessibility-conscious products. UX
impact is also high for power users.
