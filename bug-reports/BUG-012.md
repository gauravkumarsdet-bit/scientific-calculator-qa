# BUG-012 — Display uses `disabled` instead of `readonly`

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-012                                            |
| **Status**   | Open                                               |
| **Severity** | Medium                                             |
| **Priority** | P2                                                 |
| **Component**| Accessibility — display element                    |

## Summary

The `<input id="display">` element carries the `disabled` attribute.
On a disabled form control, screen readers do not announce the value
and the element is removed from the keyboard tab order. The intended
behaviour ("user cannot edit, but assistive technologies can still
read") is what `readonly` provides.

## Steps to reproduce

1. Open the calculator with a screen reader (VoiceOver / NVDA / JAWS).
2. Tab through the page.

## Expected behaviour

The display value is announced and is part of the tab order
(or at minimum is reachable & readable as a live region — see
BUG-014).

## Actual behaviour

The display is silently skipped by both screen readers and tab focus.

## Evidence

| Type | Path |
|--|--|
| Failing test | `tests/ux/test_ux_and_accessibility.py::TestDisplayAccessibility::test_display_should_be_readonly_not_disabled` |

## Suggested fix

```diff
- <input type="text" id="display" disabled>
+ <input type="text" id="display" readonly>
```

## Severity justification

**Medium.** WCAG 2.1 success criteria 4.1.2 (Name, Role, Value)
implicitly require that controls communicate their state.
A disabled output that is the *primary* output of the application is
a real accessibility problem, but does not affect sighted-keyboard or
mouse users.
