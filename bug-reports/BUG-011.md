# BUG-011 — Division by zero leaks `Infinity` / `NaN` instead of `Error`

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-011                                            |
| **Status**   | Open                                               |
| **Severity** | Medium                                             |
| **Priority** | P2                                                 |
| **Component**| `evaluateExpression` — arithmetic                  |

## Summary

`a / 0` and `0 / 0` should display `Error`. Today they leak the JS
literals `Infinity` / `-Infinity` / `NaN` into the UI (or, in
combination with BUG-003's operand-swap, even `0`).

## Steps to reproduce

1. Open the calculator.
2. Press `5`, `÷`, `0`, `=`.

## Expected behaviour

Display shows `Error`.

## Actual behaviour

| Input | Got | Expected |
|---|---|---|
| `5/0`     | `0` (cascade of BUG-003) | `Error` |
| `0/0`     | `NaN`      | `Error` |
| `(0-5)/0` | `Infinity` (cascades of BUG-001, BUG-003) | `Error` |

## Evidence

| Type | Path |
|--|--|
| Failing tests | `tests/edge_cases/test_edge_cases.py::TestDivisionByZero::*` |

## Root cause hypothesis

The arithmetic step performs `a / b` directly without checking
`b === 0`. JavaScript `5 / 0` returns `Infinity` rather than throwing,
so the leak is silent.

## Suggested fix

The single-line normalisation suggested in BUG-006 / BUG-008 — turn
non-finite results into `Error` after evaluation — fixes this defect
too:

```js
display.value = Number.isFinite(r) ? r : 'Error';
```

If a more granular UX is preferred, distinguish `Infinity` →
`"Cannot divide by zero"` from generic `NaN` → `"Error"`.

## Severity justification

**Medium.** Real arithmetic concern (and one a junior developer
will hit), but safe in the sense that the user is unlikely to mistake
`Infinity` for a valid answer. Easy to fix.
