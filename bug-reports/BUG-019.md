# BUG-019 — Pressing a digit after `=` appends to the result instead of starting a new calculation

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-019                                            |
| **Status**   | Open                                               |
| **Severity** | High                                               |
| **Priority** | P1                                                 |
| **Component**| UX / state machine                                 |

## Summary

After `=` is pressed and a result is shown, pressing a digit *appends*
that digit to the result rather than starting a fresh calculation.
Every other calculator (desktop OS, mobile OS, Google search, scientific
calculator apps) resets the display in this scenario; failing to do so
breaks user expectations and produces silently-wrong results.

## Steps to reproduce

1. Open the calculator.
2. Press `7`, `+`, `2`, `=`. Display shows `9`.
3. Press `5`.

## Expected behaviour

Display shows `5` (the user has clearly started a new calculation).

## Actual behaviour

Display shows `95` — the `5` is appended to the previous result.

| Type | Path |

## Suggested fix

Track a tiny piece of UI state — "last action was `=`" — and clear the
display on the *next* digit / decimal:

```js
let justEvaluated = false;

function append(value) {
  if (justEvaluated) {
    display.value = '';
    justEvaluated = false;
  }
  display.value += value;
}
function calculate() {
  …
  justEvaluated = true;
}
```

(Operator presses after `=` should keep the result and append the
operator — i.e. continue the calculation. The above rule, applied
only to digit / decimal appends, achieves both behaviours.)

## Severity justification

**High.** This is functional, not cosmetic — the user does *not*
realise the digits have been appended (a glance at the screen sees
"5" followed by another digit and assumes the next typing started
fresh). Result: wrong answers, silently. Severity matches a typical
"calculator computes the wrong number" defect.
