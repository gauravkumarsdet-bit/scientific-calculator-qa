# BUG-002 — Digit `3` button appends `0` to the display

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-002                                            |
| **Status**   | Open                                               |
| **Severity** | Critical                                           |
| **Priority** | P0                                                 |
| **Component**| UI — button wiring                                 |
| **Reporter** | Gaurav Kumar (QA)                                  |
| **Reported** | 2026-05-07                                         |

## Summary

Pressing the button visibly labelled `3` appends the character `'0'`
to the display, not `'3'`. Every calculation that involves the digit 3
is therefore silently wrong — typically yielding `0` for what should
be a non-zero answer.

## Steps to reproduce

1. Open the calculator.
2. Press `C` (clear).
3. Press `3`.
4. Observe the display.

## Expected behaviour

Display shows `3`.

## Actual behaviour

Display shows `0`.

## Evidence

| Type | Path |
|--|--|
| Failing test | `tests/functional/test_button_label_integrity.py::test_digit_button_inserts_its_own_label[digit=3]` |
| Cascading failures | `1/3 → 0`, `9/3 → 0`, `13+1 → 11`, etc. |
| Screenshot | `reports/screenshots/test_digit_button_inserts_its_own_label*.png` |

## Root cause hypothesis

In `index.html`:

```html
<button onclick="append('0')">3</button>
```

A copy-paste from the row above. All other digit buttons are wired
correctly (verified by parametrising the test over digits 0-9 — only
`3` fails).

## Suggested fix

```diff
- <button onclick="append('0')">3</button>
+ <button onclick="append('3')">3</button>
```

## Severity justification

**Critical.** Like BUG-001 the symptom is a *plausible* wrong answer —
typically `0` for any expression in which `3` is a divisor or
multiplicand — and there is no workaround. Users will lose trust the
moment they spot it; until then they will silently rely on bad numbers.

## Notes / related bugs

* Interacts heavily with BUG-003 (division operand-swap). For example,
  `1/3` is entered as `1/0`, swap → `0/1 = 0`. Untangling the two
  defects required parametrising digit-entry tests independently of
  any operator (see `test_digit_button_inserts_its_own_label`).
