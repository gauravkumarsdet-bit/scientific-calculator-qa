# BUG-004 — Multiple decimal points in a number are silently truncated

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-004                                            |
| **Status**   | Open                                               |
| **Severity** | High                                               |
| **Priority** | P1                                                 |
| **Component**| Lexer / `tokenize`                                 |

## Summary

The lexer accepts an arbitrary run of `[\d.]` characters as a single
number token, so inputs like `1.2.4` and `1..5` — neither of which is
a valid number — are accepted, fed to `parseFloat`, and silently
truncated. The user gets a wrong-but-plausible result with no
indication that part of their input was discarded. This is a
silent-data-loss defect.

## Steps to reproduce

1. Open the calculator.
2. Press `1`, `.`, `2`, `.`, `4`.
3. Press `=`.

## Expected behaviour

Either reject the second `.` at the input layer, or produce `Error`
on evaluation. The currently-existing `Error` pathway is a perfectly
good destination for this case.

## Actual behaviour

Display shows `1.2`. The `.4` is silently dropped.

| Input  | Expected | Actual |
|--------|----------|--------|
| `1.2.4` | `Error` (or rejection at input) | `1.2` |
| `1..5`  | `Error` | `1` |

## Evidence

| Type | Path |
|--|--|
| Failing tests | `tests/functional/test_digits_and_decimal.py::TestDecimalPoint::test_multiple_decimal_points_must_be_rejected`, `…::test_consecutive_decimal_points_must_be_rejected` |

## Root cause hypothesis

In `tokenize`, the inner loop accumulates while
`/[\d.]/.test(expr[i])` — there is no state tracking whether a `.`
has already been seen for the current token. `parseFloat('1.2.4')`
silently returns `1.2`.

## Suggested fix

In `tokenize`, for each numeric token, allow at most one `.`:

```js
let seenDot = false;
while (i < expr.length && /[\d.]/.test(expr[i])) {
  if (expr[i] === '.') {
    if (seenDot) { /* throw or return 'Error' token */ }
    seenDot = true;
  }
  numStr += expr[i++];
}
```

Alternatively, disable the `.` button when the *current* number token
already contains one — that prevents the bad input being entered at
all.

## Severity justification

**High.** Silent data loss is among the worst calculator bugs:
the user has no signal at all that their input was malformed and that
the answer is therefore unreliable. Probability of occurrence is low
(typing two dots is an obvious error), but consequence is severe when
it does happen.
