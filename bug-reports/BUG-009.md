# BUG-009 — Parser silently drops everything after a closing `)` mid-expression

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-009                                            |
| **Status**   | Open                                               |
| **Severity** | High                                               |
| **Priority** | P1                                                 |
| **Component**| Parser — recursive-descent expression evaluator    |

## Summary

When a parenthesised sub-expression `(…)` appears anywhere except as
the right-most token, every token after the closing `)` is silently
discarded. The user sees a wrong-but-plausible number with no signal
that part of their expression was thrown away. This is the kind of
silent-arithmetic bug that ships and rots in a customer's spreadsheet
for years.

## Steps to reproduce

1. Open the calculator.
2. Press `(`, `2`, `+`, `5`, `)`, `*`, `4`, `=`.
3. Observe the display.

## Expected behaviour

`(2 + 5) * 4 = 28`. Display shows `28`.

## Actual behaviour

Display shows `7`. The `* 4` is silently dropped.

| Input | Expected | Actual |
|---|---|---|
| `(2+5)*4`     | `28` | `7` |
| `(2+5)+1`     | `8`  | `7` |
| `(2+5)*4+1`   | `29` | `7` |
| `1+(2+5)*4`   | `29` | `8` |
| `(1+1)*(1+1)` | `4`  | `2` |

## Evidence

| Type | Path |
|--|--|
| Failing tests | `tests/edge_cases/test_edge_cases.py::TestParserTruncatesAfterClosingParen::*` |
| Working cases | `tests/edge_cases/test_edge_cases.py::TestParenthesesAlone::*` (closing paren is rightmost — passes) |

## Root cause hypothesis

In the recursive-descent parser, the `factor` rule consumes
`(...)` and *returns immediately* after the closing paren:

```js
if (_0x80f5f[_0x755c] === '(') {
  _0x755c++;
  const inner = _0x01183a();          // expr()
  if (_0x80f5f[_0x755c] === ')') _0x755c++;
  return inner;                        // <- correct
}
```

That part is correct. The bug is one level up: the loop in `expr()` /
`term()` reads `_0x80f5f[_0x755c]` to decide whether to continue, but
the "expr" start is invoked at top of `calculate` only once and never
re-entered after a paren factor returns. So tokens after the paren
sit on the input stack but are never consumed.

A working implementation runs `term()` and `expr()` *as loops*:

```
expr  := term ( ('+'|'-') term )*
term  := factor ( ('*'|'/') factor )*
```

The current implementation appears to read the operator-after-factor
condition incorrectly when the factor was a paren expression. (The
exact line is hard to pinpoint in the obfuscated source — a developer
reading the deobfuscated grammar will likely spot it in 5 minutes.)

## Suggested fix

Re-implement `expr` / `term` as straightforward loops over operands &
operators, returning only when the next token is not an operator
recognised at that level. Pseudocode:

```js
function expr() {
  let v = term();
  while (peek() === '+' || peek() === '-') {
    const op = consume();
    const rhs = term();
    v = (op === '+') ? v + rhs : v - rhs;
  }
  return v;
}
```

(This is exactly the structure the project's QA suite uses in
`src/utils/math_oracle.py` for the test oracle, which evaluates the
same grammar correctly.)

## Severity justification

**High.** Parenthesised expressions are a normal use case for a
"scientific" calculator. The defect is silent and the wrong answer is
plausible (`(2+5)*4` returning `7` looks superficially similar to the
real answer of `28` for someone glancing at the screen). Combined with
the silent nature, this should be P1.

## Notes / related bugs

* The test `tests/edge_cases/test_edge_cases.py::TestParenthesesAlone`
  verifies this defect *does not* fire when `)` is the rightmost
  token, which gives the developer a clear repro / non-repro pair.
