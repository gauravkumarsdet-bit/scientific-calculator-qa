# BUG-003 — Division operands are swapped in the parser

| Field        | Value                                              |
| ------------ | -------------------------------------------------- |
| **ID**       | BUG-003                                            |
| **Status**   | Open                                               |
| **Severity** | Critical                                           |
| **Priority** | P0                                                 |
| **Component**| Parser / `evaluateExpression`                      |
| **Reporter** | Gaurav Kumar (QA)                                  |
| **Reported** | 2026-05-07                                         |

## Summary

The expression evaluator computes `b / a` when the user enters `a / b`.
The bug is in the recursive-descent `term` rule: the local variable
holding the running accumulator is divided *into* the right-hand
operand, instead of being divided *by* it. As a consequence every
non-trivial division is silently wrong; the calculator is unusable for
any workflow involving division.

## Steps to reproduce

1. Open the calculator.
2. Press `8`, then `÷`, then `2`, then `=`.
3. Observe the display.

## Expected behaviour

`8 ÷ 2 = 4` — display shows `4`.

## Actual behaviour

Display shows `0.25` (= `2 / 8`).

Verified across multiple inputs:

| Input | Got | Expected |
|---|---|---|
| `8/2` | `0.25` | `4` |
| `7/1` | `0.142857…` | `7` |
| `100/4` | `0.04` | `25` |
| `4/2` | `0.5` | `2` |

## Root cause hypothesis

Inside the obfuscated `term` rule (`function _0x8329bc()`):

```js
const _0x3_0xc58 = _0x80f5f[_0x755c++];   // operator
const _0x692a    = _0x4c3dcb();           // RHS
if (_0x3_0xc58 === '*') _0x831edg = _0x692a * _0x831edg;
else                    _0x831edg = _0x692a / _0x831edg;  // BUG: should be _0x831edg / _0x692a
```

Multiplication happens to be commutative so the swap is invisible
there, which is why only division surfaces the defect.

## Suggested fix

```js
- else _0x831edg = _0x692a / _0x831edg;
+ else _0x831edg = _0x831edg / _0x692a;
```

## Severity justification

**Critical.** Every division produces wrong-but-plausible results; users
have no way to detect it without an external reference, and several
common inputs (e.g. `5/0`) produce `0` instead of `Infinity`, masking
even the divide-by-zero pathway.
